# Phase 6 - 6.5 Pacing 调优、验证证据与 Taste 签署

**阶段**: 6.5  
**优先级**: P0  
**对应问题**: 15 分钟口径缺乏节奏参数与证据 + Phase 6 发布收口

---

## 1. 阶段目标

把“15 分钟 5 层”从文案口径变成可调优、可验收、可签署的节奏目标，并统一完成 Phase 6 发布收口。

完成标准：

1. Normal 中位数 run 时长稳定在 `12~18 min`。
2. Normal P90 `<= 20 min`。
3. 15 分钟目标具备参数表、样本报告与手动录像证据。
4. 技术门禁与 taste 门禁同时通过。

---

## 2. 当前问题

1. 当前只有 `elapsedMs` 结算记录，没有明确 pacing 参数表。
2. 没有 floor-level time target，也没有节奏偏移报警。
3. 即便 choice / spike / rhythm / feedback 做完，没有统一 evidence pack 也无法签署。

---

## 3. 范围与非目标

### 3.1 范围

1. floor-level pacing 指标与参数表。
2. Normal / Hard / Nightmare 节奏目标区间。
3. simulator + 手动录像双证据。
4. Phase 6 smoke matrix。
5. feature flag / rollback 策略。
6. 最终 taste sign-off。
7. 资源链路与 fallback 收口。

### 3.2 非目标

1. 不在 6.5 再引入新的核心玩法系统。
2. 不在 6.5 扩展楼层数或职业数。
3. 不做新的 content 大爆发。

---

## 4. 技术方案

### 4.1 Pacing 参数表

建立并冻结以下调参手柄：

1. 地图尺寸
2. 怪物数量
3. 清层阈值
4. 楼梯显现时机
5. 玩家移动速度
6. 商店 / 事件密度
7. core skill cadence 目标

### 4.2 节奏验证双轨制

1. simulator：
   - 结构偏差检测
   - 参数回归
2. 手动 run：
   - 真实体感确认
   - 录像归档
   - 高压场景复盘

任何节奏签署都必须同时具备两类证据。

### 4.3 Smoke Matrix

至少覆盖：

1. Normal story run
2. Hard story run
3. Nightmare story run
4. rare / build / boss 峰值 run
5. 三职业起步深度 run
6. trade-off item run
7. buff / damageType / synergy 合同 run

### 4.4 Release Closure

1. 输出最终 evidence 包。
2. 明确 known issues。
3. 配置 feature flag 与回滚表。
4. 形成 Phase 6 签署清单。
5. 若 Phase 6 产生资源增量，补齐视觉/音频 manifest、运行目录与 fallback 证据。

## 6. 验证方案

### 6.1 自动化

```bash
pnpm quality:precheck
pnpm check
pnpm test
pnpm --filter @blodex/core test
pnpm --filter @blodex/game-client test
pnpm --filter @blodex/game-client i18n:check
pnpm --filter @blodex/game-client css:check
pnpm assets:audio:compile
pnpm assets:audio:validate
pnpm assets:validate
pnpm check:architecture-budget
pnpm ci:check
```

新增重点：

1. pacing table 读取与报告生成测试。
2. rhythm / spike / choice / feedback 汇总一致性测试。
3. sign-off checklist 完整性检查。
4. 资源 manifest 与运行时 key 一致性检查。
5. buff / damageType / synergy 收口检查。

### 6.2 手动冒烟

1. Debug fast-forward：
   - floor 1~5 全链路 choice / spike / rhythm / reward 检查
2. Non-cheat normal run：
   - 至少 3 局完整 run
3. Class parity：
   - warrior / ranger / arcanist 各 1 局
4. Build formed 验证：
   - lifesteal / crit / aoe / dot 路线至少各 1 次
5. Trade-off 装备验证：
   - 至少 2 次出现“更强但更痛”的装备替换抉择
6. Rhythm before/after 验证：
   - 对比 6.0 baseline 与最终结果的平均无主动输入间隔、技能释放频率与自动攻击占比
7. 资源闭环验证：
   - 检查新增 cue 在真实包内可加载、可播放、可降级
8. Runtime effect 验证：
   - `war_cry / shadow_step / frost_nova` 至少各 1 次
   - `physical / arcane` 至少各 1 组
   - 至少 1 次 `synergy activated` 反馈

---

## 7. 出口门禁

1. Normal 中位数 run 时长稳定在 `12~18 min`。
2. Normal P90 `<= 20 min`。
3. 技术门禁与 taste 门禁同时通过。
4. Phase 6 evidence 包完整可审阅。
5. 如存在资源增量，视觉/音频校验链全部通过。

---

## 8. 风险与止损

1. 风险：仅凭 simulator 通过，但手感仍不对。  
   止损：必须保留手动录像与人工复盘。
2. 风险：为压时长而牺牲心跳时刻。  
   止损：pacing 调优不得回退 choice / spike / feedback 合同。
3. 风险：Phase 6 收口时 known issue 不透明。  
   止损：发布前必须输出明确清单，禁止口头略过。

---

## 9. Phase 6 完成定义（DoD）

只有以下条件全部满足，Phase 6 才允许关门：

1. 每层至少 1 次关键构筑选择。
2. 每 2 层至少 1 次达到最低幅度标准的战力跃迁。
3. rare / build formed / boss reward 都有玩家可见反馈。
4. 战斗节奏合同达标，技能不再只是偶发点缀。
5. 已进入内容定义的 buff/debuff 在 runtime 中真实生效。
6. `physical / arcane` 已形成最小差异化。
7. `damageOverTime` 已完成语义收口。
8. `arcanist` 与 warrior / ranger 的起步深度差距显著收敛。
9. 至少一组核心装备提供真实 trade-off。
10. 至少一类 synergy 激活在玩家侧有明确反馈。
11. 15 分钟目标有真实样本报告支撑。
12. dungeon Host Port 的 `any` 债被实质收口。
13. 技术门禁与 taste 门禁同时通过。
14. 若 Phase 6 产生新增视觉或音频资源，对应 manifest 校验与 fallback 检查全部通过。
