# Phase 6 - 6.0 Cue 资源缺口审计

**日期**: 2026-03-06  
**对应阶段**: `6.0 baseline observability and architecture closure`  
**核查范围**:

1. `apps/game-client/src/systems/SFXSystem.ts`
2. `apps/game-client/src/systems/VFXSystem.ts`
3. `apps/game-client/src/systems/feedbackEventRouter.ts`
4. `assets/source-prompts/audio-plan.yaml`
5. `docs/plans/phase6/2026-03-06-phase6-6.0-baseline-observability-and-architecture-closure.md`

---

## 1. 直接结论

Phase 6.0 已经把 `player_facing_choice / power_spike / build_formed / rare_drop_presented / boss_reward_closed / combat_rhythm_window / synergy_activated`
变成可观测事件，但 **并没有同步补齐对应的表达层 cue 资源**。

本轮审计结论如下：

1. `6.0` 不需要新增硬性美术资源；现有 `VFXSystem` 已足够支撑 6.3 的第一版表达层。
2. `6.3` 至少需要补一组最小音频 cue；当前 `SFXSystem` 和 `audio-plan.yaml` 都没有对应 key。
3. “rare drop / build formed / boss reward / equipment compare” 的视觉第一版应优先复用现有 code-driven VFX，而不是扩成新的 UI frame / sprite 项目。
4. 若后续要做更强表现，建议作为 `6.3 optional enhancement`，不要回流到 6.0 或阻塞 6.1。

---

## 2. 当前资源与能力现状

### 2.1 已有音频覆盖

`SFXSystem` 当前已覆盖：

1. `combat hit / crit / death`
2. `skill use`
3. `consumable use`
4. `event spawn`
5. `merchant open / buy / fail`
6. `floor enter / biome enter / biome ambient`
7. `boss phase`
8. `hazard trigger`
9. `level up`

缺口：

1. 没有 `rare drop` 专属 cue
2. 没有 `build formed` 专属 cue
3. 没有 `boss reward` 专属 cue
4. 没有 `equipment compare` 专属 cue

`feedbackEventRouter` 当前也没有上述四类 presentation input，因此即便 telemetry 已经落地，玩家侧仍不会得到对应的 SFX 路由。

### 2.2 已有视觉覆盖

`VFXSystem` 当前已具备：

1. `flashTarget`
2. `floating text`
3. `camera flash / shake`
4. `ellipse ring`
5. `skill cast ring`
6. `level up pulse`
7. `boss phase` 的高优先级瞬时表现

这意味着：

1. `rare drop` 可以先用 `camera flash + floating text + ring`
2. `build formed` 可以先用 `floating text + center pulse`
3. `boss reward` 可以先用 `flash + ring + panel focus`
4. `equipment compare` 可以先用 `panel highlight + short flash`

因此第一版 **不需要** 新增 sprite sheet、特殊边框图或大图插画。

## 3. 资源冻结边界

### 3.1 6.3 必需新增的最小音频集

建议冻结为以下 4 个 key：

1. `ui_loot_rare_drop_01`
2. `ui_build_formed_01`
3. `ui_boss_reward_open_01`
4. `ui_equipment_compare_open_01`

推荐事件映射：

1. `rare_drop_presented -> ui_loot_rare_drop_01`
2. `build_formed -> ui_build_formed_01`
3. `boss reward panel open -> ui_boss_reward_open_01`
4. `equipment compare prompt open -> ui_equipment_compare_open_01`

说明：

1. `boss_reward_closed` 是 6.0 的 telemetry 事件，用来记录玩家代理权闭环；
2. `ui_boss_reward_open_01` 是 6.3 的 presentation cue，二者不冲突；
3. 6.3 应在 UI 展示层增加 “panel opened” 的 route，而不是为了音频去重写 6.0 telemetry 事件名。

### 3.2 6.3 默认复用的视觉集

默认复用而不新建资源：

1. `camera flash`
2. `camera shake`
3. `floating text`
4. `ellipse ring`
5. `panel tint / focus pulse`

### 3.3 仅作为可选增强的视觉资产

这些都不应作为 `6.3` 入口门槛：

1. rare / unique 专属面板边框
2. build formed 专属 banner 图
3. boss reward 专属插画背景
4. equipment compare 专属 frame sprite

---

## 4. 对 6.3 的实施约束

1. 新增音频必须走 `audio-plan -> audio-manifest -> assets:audio:validate`。
2. 第一版视觉必须优先走现有 `VFXSystem`，避免 scope 膨胀到美术制作链。
3. 若 6.3 需要增加新的 feedback router input，应只补 presentation route，不得回改 6.0 telemetry 合同。
4. 若音频资源未及时到位，6.3 必须提供 fallback：
   - rare drop 可临时复用 `level_up` 或 `merchant_open` 类 UI cue；
   - build formed 可临时复用 `level_up`；
   - boss reward / compare 可临时静默，但必须保留视觉表现。

---

## 5. 审计结论

6.0 已经把“什么时候该响、什么时候该亮”定义清楚了；  
6.3 只需要把 **四个音频 cue** 和少量 presentation route 补上，就能把 heartbeat 表达层接通。

因此，Phase 6 在资源侧的正确策略是：

1. `6.0` 冻结需求边界；
2. `6.3` 补最小音频集；
3. 视觉继续以代码驱动表现为主；
4. 美术增强放在后续可选项，而不是当前阻塞项。
