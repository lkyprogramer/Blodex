## 1. 阶段目标

把 Phase 6 要解决的问题从评论语义转成 runtime 可观测对象，并同步收口会阻碍后续迭代的 Host Port 边界。

完成标准：

1. Phase 6 关键体验项具备统一事件字典与统计口径。
2. baseline 报告可稳定输出：
   - 每层 choice 次数
   - 每局 spike 次数
   - build formed 次数
   - 平均 run 时长
   - 技能释放频率
   - 自动攻击 / 技能伤害占比
   - 平均无主动输入间隔
3. `RealBalanceSimulator` 或等价工具至少具备一版“带技能使用”的观测路径。
4. 核心 dungeon runtime module 不再依赖裸 `Record<string, any>` Host。
5. 6.3 所需资源缺口被明确归类，不再在实现阶段临时决定。
6. buff/debuff 与 damageType 的断点已被量化，能够支撑 6.1 的系统补完。

---

## 2. 当前问题

1. 现在能描述“战斗节奏差”“跃迁不够强”，但缺少统一的 runtime 指标，无法做 before/after 对比。
2. `TasteRuntimePortHub` 和 run summary 已能承载部分诊断，但没有覆盖 `combat rhythm / spike amplitude / player-facing choice`。
3. 现有 `RealBalanceSimulator` 基本看不见技能循环变化，不足以为 Phase 6 的节奏改造签署。
4. `scenes/dungeon/*` 仍残留 `Record<string, any>` Host Port，会放大后续阶段改造风险。
5. 当前 `SFXSystem` 只覆盖 combat / skill / event / merchant / floor / biome / boss phase / level up；没有 rare drop / build formed / boss reward / equipment compare 的专属 audio key。
6. 当前 `VFXSystem` 已具备 flash / shake / floating text / ellipse 这类 code-driven 表现能力，但“是否需要新 sprite / UI frame”尚未形成归档结论。
7. `resolveSkill()` 虽然已经产出 `buffsApplied`，但实机 runtime 没把它写回 player / monster state。
8. `damageType` 已进入 skill def 与 combat event，但没有进入可区分的结算规则或 baseline 报告。

## 4. 技术方案

### 4.1 Phase 6 事件字典冻结

新增并冻结以下事件：

1. `player_facing_choice`
2. `power_spike`
3. `build_formed`
4. `rare_drop_presented`
5. `boss_reward_closed`
6. `combat_rhythm_window`
7. `synergy_activated`

要求：

1. 所有事件都必须能进入 `Taste / Feedback / RunLog` 三条链中的至少两条。
2. 事件 payload 禁止只传文案；必须传可复盘字段。

### 4.2 Baseline 报告

输出至少两类基线：

1. `story-run-baseline`
   - 每层关键选择
   - 每局 spike 次数
   - 平均时长
2. `combat-rhythm-baseline`
   - skill casts / 30s
   - auto-attack damage share
   - mana dry window
   - average no-input gap
3. `runtime-effect-baseline`
   - `war_cry / guaranteed_crit / frost_slow` 的 apply 次数与 uptime
   - `physical / arcane` 伤害占比与结算占比
   - synergy 激活次数与首次出现楼层

### 4.3 Cue Asset Gap Audit

1. 盘点 rare / build / boss / compare 四类 cue 当前是否已有可复用资源。
2. 输出三类结论：
   - 6.3 必需新增的音频资源
   - 现有 `VFXSystem` 可直接承载的视觉表现
   - 仅作为可选增强的视觉资产
3. 不在 6.0 直接生产资源，但必须把资源边界写清，避免 6.3 临时扩 scope。

### 4.4 Runtime Effect Gap Audit

1. 追踪 `resolveSkill().buffsApplied -> player.activeBuffs / monster debuffs / deriveStats` 的断点。
2. 明确 `MonsterState` 缺少 debuff 承载字段后，`frost_nova` 等效果无法落盘的事实。
3. 识别 `damageType` 在结算层仅作标签、不参与规则分流的位置。
4. 为 6.1 输出最小补完边界，而不是在 6.0 直接扩成完整元素体系。

### 4.5 Balance Tooling 补线

1. `RealBalanceSimulator` 增加最小技能使用路径。
2. 在该能力完成前，禁止用它单独为 `combat tempo` 签署。
3. 保留 heuristic / real 差异报告，但新增 “tempo evidence incomplete” 状态。

### 4.6 Host Port 边界收口

1. 清理 `scenes/dungeon/*` 中阻碍 Phase 6 的核心 `[key: string]: any`。
2. 按职责拆分最小 Host Port：
   - telemetry host
   - progression host
   - reward host
   - feedback host
3. `DungeonScene` 仅做装配，不再成为隐式全局状态容器。

## 8. 风险与止损

1. 风险：为观测补线引入过多埋点噪音。  
   止损：先做领域级事件，不做细粒度逐帧日志。
2. 风险：simulator 改动过大拖慢主线。  
   止损：只补“最小技能使用路径”，不在 6.0 追求完整仿真。
3. 风险：类型化改造导致编译链式报错。  
   止损：按模块批次提交，先收口最关键 Host Port。
