# Phase 6 - 6.3 架构预算门禁后续收口

**日期**: 2026-03-06  
**目的**: 解决 `pnpm check:architecture-budget` 因 `6.3` 反馈层与比较提示接线触发的历史大文件预算失败，同时保持门禁对后续回流与体量反弹仍然有效。

---

## 1. 直接结论

本次收口不放宽目标预算。

调整的是门禁语义：

1. **目标预算** 继续保留：
   - `DungeonScene.ts <= 2600 lines / 90 methods`
   - `HudContainer.ts <= 1100 lines / 60 methods`
2. **临时债务上限** 冻结为本次 6.3 落地后的实测值：
   - `DungeonScene.ts <= 4286 lines / 92 methods`
   - `HudContainer.ts <= 1186 lines`
3. CI 规则改为：
   - 非白名单文件：仍按目标预算硬性阻断。
   - 命中 debt ceiling 的历史文件：允许高于目标预算，但**不允许再超过本次冻结值**。
4. 因此 `check:architecture-budget` 继续承担 no-regression gate，而不是被 6.3 的历史债直接常驻卡死。

---

## 2. 为什么需要这次收口

6.3 新增的是：

1. `HeartbeatFeedbackRuntime`
2. `EquipmentComparePrompt`
3. feedback router / SFX / VFX / HUD overlay 接线

这些实现遵循了“优先复用现有 `VFXSystem` / `UIManager` / `HudContainer`”的阶段策略，但现实结果是：

1. `DungeonScene.ts` 作为场景壳仍承担了新的反馈 runtime 接线与高压峰值检测。
2. `HudContainer.ts` 作为现有 HUD 容器继续吸收了 compare prompt 与 toast overlay。
3. 若继续用“只看目标预算”的单层门禁，CI 会把 6.3 的有效落地和历史大文件债务混为一谈。

这会让门禁重新退化成噪音，而不是治理工具。

---

## 3. 本次冻结的门禁口径

### 3.1 目标预算

1. `DungeonScene.ts <= 2600 lines / 90 methods`
2. `MetaMenuScene.ts <= 1200 lines / 85 methods`
3. `Hud.ts <= 300 lines / 25 methods`
4. `HudContainer.ts <= 1100 lines / 60 methods`

### 3.2 临时债务上限

仅对当前仍存在历史债的文件启用：

1. `DungeonScene.ts <= 4286 lines / 92 methods`
2. `HudContainer.ts <= 1186 lines`

说明：

1. 这是 **debt ceiling**，不是新的目标预算。
2. 一旦文件压回目标预算以内，应移除对应 debt ceiling。
3. 后续若再继续膨胀，CI 必须重新 fail，而不是自动跟涨 ceiling。

---

## 4. 对 6.3 的约束

这次 follow-up 不意味着可以继续把反馈层逻辑回流主场景或 HUD 容器。

后续 PR 必须遵守：

1. 新的 heartbeat / compare / overlay 逻辑优先进入 `scenes/dungeon/feedback/*`、`ui/components/*` 或 presenter，不再继续加厚 `DungeonScene.ts` 与 `HudContainer.ts`。
2. 若修改 `DungeonScene.ts`，必须保证：
   - 不突破 `4286 lines / 92 methods` 的临时债务上限；
   - 优先通过模块抽取抵消新增体量。
3. 若修改 `HudContainer.ts`，必须保证：
   - 不突破 `1186 lines` 的临时债务上限；
   - 新 overlay 优先做成独立 component / presenter，而不是继续把渲染逻辑塞回容器。

---

## 5. 出口条件

本 follow-up 关闭标准：

1. `pnpm check:architecture-budget` 可在当前分支通过。
2. `DungeonScene.ts` 与 `HudContainer.ts` 的历史债状态在 CI 输出中可见。
3. 门禁能阻止后续把这两处文件继续推高。
4. 脚本与文档口径一致。

---

## 6. 后续待办

真正的根治仍然是继续拆分：

1. 将 `DungeonScene` 中与 heartbeat / pressure peak / compare queue 相关的 runtime 继续外提。
2. 将 `HudContainer` 中 compare prompt / tooltip / toast 的复用展示逻辑继续抽成更薄的 view adapter。
3. 当 `DungeonScene.ts <= 2600 / 90`、`HudContainer.ts <= 1100 / 60` 后，删除本文件对应的 debt ceiling。
