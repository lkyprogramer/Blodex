# Phase 6 - 6.2 架构预算门禁后续收口

**日期**: 2026-03-06  
**目的**: 解决 `pnpm check:architecture-budget` 因历史 `DungeonScene.ts` 行数债务而常驻阻塞的问题，同时保持门禁对新增回流与体量回退仍然有效。

---

## 1. 直接结论

本次收口不放宽 `DungeonScene.ts` 的目标预算。

调整的是门禁语义：

1. **目标预算** 继续保留：`DungeonScene.ts <= 2600 lines / 90 methods`。
2. **临时债务上限** 冻结为本次收口后的实测值：`4165 lines`。
3. CI 规则改为：
   - 非白名单文件：仍按目标预算硬性阻断。
   - `DungeonScene.ts`：允许高于目标预算，但**不允许超过临时债务上限**。
4. 因此 `check:architecture-budget` 从“被历史债务永久卡死”变成“历史债务可见、未来回退不可接受”的 no-regression gate。

---

## 2. 为什么需要这次收口

当前仓库已有两组事实同时存在：

1. 文档长期要求新逻辑不要回流 `DungeonScene.ts`。
2. `origin/main` 已经处于 `DungeonScene.ts > 2600` 的历史债状态。

如果继续用“超过目标预算即 fail”的单层规则，结果是：

1. 当前主干本身无法通过门禁。
2. 任何功能分支都会被历史债务直接拦住。
3. CI 无法区分“我这轮让 God class 更糟了”与“仓库已有存量债”。

这会让门禁失去治理价值，只剩噪音。

---

## 3. 本次冻结的门禁口径

### 3.1 目标预算

1. `DungeonScene.ts <= 2600 lines / 90 methods`
2. `MetaMenuScene.ts <= 1200 lines / 85 methods`
3. `Hud.ts <= 300 lines / 25 methods`
4. `HudContainer.ts <= 1100 lines / 60 methods`

### 3.2 临时债务上限

仅对仍存在历史债的文件启用：

1. `DungeonScene.ts <= 4165 lines`

说明：

1. 这是 **debt ceiling**，不是新的目标预算。
2. 只有在文件已被 allowlist 标记且仍高于目标预算时，才使用该上限。
3. 一旦文件压回目标预算以内，应移除对应 debt ceiling。

---

## 4. 执行语义

门禁脚本现在分两层判断：

1. 先按目标预算检查。
2. 若超出目标预算，再看是否命中显式 debt ceiling。

结果分类：

1. `Pass`：低于目标预算。
2. `Warning`：高于目标预算，但仍在 debt ceiling 以内。
3. `Fail`：超过 debt ceiling，或非白名单文件超过目标预算。

---

## 5. 对 Phase 6 的约束

这次收口不是允许 `DungeonScene.ts` 再继续膨胀。

Phase 6 后续阶段必须遵守：

1. 新功能默认进入 `scenes/dungeon/*` 模块，不回流主场景。
2. 若修改 `DungeonScene.ts`，必须保证：
   - 不突破 `4165 lines` 的临时债务上限；
   - 方法数不突破 `90`；
   - 优先通过模块抽取抵消新增体量。
3. 后续若再压低 `DungeonScene.ts`，应同步收紧 debt ceiling，而不是保留旧值。

---

## 6. 出口条件

本 follow-up 关闭标准：

1. `pnpm check:architecture-budget` 可在当前分支通过。
2. `DungeonScene.ts` 的历史债状态在 CI 输出中可见。
3. 门禁能阻止后续把 `DungeonScene.ts` 再推高。
4. 文档与脚本口径一致。

---

## 7. 后续待办

真正的根治仍然是继续拆分 `DungeonScene.ts`。

建议下一步单独立项：

1. 抽离更多 scene-only adapter 到 `scenes/dungeon/*`。
2. 继续把奖励、日志、交互、HUD 计算从主场景移出。
3. 当 `DungeonScene.ts <= 2600` 后，删除本文件对应的 debt ceiling。
