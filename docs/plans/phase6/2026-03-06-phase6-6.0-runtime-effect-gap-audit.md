# Phase 6 - 6.0 Runtime Effect Gap 审计

**日期**: 2026-03-06  
**对应阶段**: `6.0 baseline observability and architecture closure`  
**核查范围**:

1. `packages/core/src/skill.ts`
2. `packages/core/src/contracts/types.ts`
3. `packages/core/src/contracts/events.ts`
4. `apps/game-client/src/scenes/dungeon/encounter/PlayerActionModule.ts`
5. `apps/game-client/src/scenes/dungeon/logging/DomainEventEffectBinder.ts`
6. `apps/game-client/src/scenes/dungeon/taste/Phase6Telemetry.ts`
7. `apps/game-client/src/scenes/DungeonScene.ts`

---

## 1. 直接结论

6.0 已经把 `buff / damageType / synergy` 从“看不见的 runtime 断点”变成了“可观测、可复盘、可继续施工”的对象，但 **并没有把它们全部做成玩法闭环**。

结论分三条：

1. `buff` 现在已进入 telemetry、run log、save/resume 契约，但仍未写回完整 runtime 状态。
2. `damageType` 已进入 skill def、combat event、telemetry baseline 和 summary，但结算层仍没有 `physical / arcane` 策略分流。
3. `synergy` 的激活证据已经进入 heartbeat、event bus、run log 和 telemetry；缺口已从“观测盲区”收敛为“玩家反馈层尚未表达”。

---

## 2. Buff / Debuff 链路现状

### 2.1 已闭环的部分

当前链路已经存在：

1. `SkillDef.effect.type === buff/debuff`
2. `resolveSkill()` 产出 `buffsApplied`
3. `PlayerActionModule` 发出 `buff:apply`
4. `Phase6Telemetry` 记录 `buffApplyCountById` 与预期 `buffUptimeMsById`
5. `RunSummary.phase6Telemetry` 与 save/resume 可以保留这些数据
6. `DomainEventEffectBinder` 能把 `buff:apply` 写入 run log

也就是说，**6.0 已经有能力回答 “这局 buff 有没有被触发、触发了几次、理论持续了多久”**。

### 2.2 未闭环的部分

当前仍存在三个断点：

1. `PlayerState.activeBuffs` 没有形成统一 runtime 消费链
2. `MonsterState` 仍没有 debuff 承载字段
3. `buff:expire` 只有事件契约，没有完整 runtime 发射与回收路径

结果：

1. `war_cry / guaranteed_crit / frost_slow` 的应用证据可见，但战斗规则侧仍不完整
2. 对怪物的 debuff 只能停留在“技能意图”与“观测记录”，没有正式状态落盘
3. 6.1 需要补的是 `BuffRuntimeClosure`，不是再补一层观测

---

## 3. DamageType 链路现状

### 3.1 已闭环的部分

当前 `damageType` 已经贯穿：

1. `SkillDef.damageType`
2. `CombatEvent.damageType`
3. `Phase6Telemetry.damageDealtByType / damageTakenByType / resolvedHitCountByType`
4. `RunSummary.phase6Telemetry`
5. run log 与 death reason 文案

因此 6.0 已经可以稳定回答：

1. `physical` 与 `arcane` 在一局里分别造成了多少伤害
2. 玩家承受了多少各类型伤害
3. 某次 run 的输出结构是偏物理还是偏奥术

### 3.2 未闭环的部分

当前结算层仍缺：

1. `physical / arcane` 差异化减伤或穿透规则
2. 怪物侧抗性 / 弱点表
3. damageType 对 build 决策的真实策略影响

结果：

1. `damageType` 已经从“纯日志标签”升级为“可观测标签”
2. 但它仍不是“玩法规则”
3. 6.1 应实现 `DamageTypeResolverV1`，先做最小分流，不要在本轮直接扩成完整元素体系

---

## 4. Synergy 链路现状

### 4.1 已闭环的部分

`synergy` 在 6.0 已完成以下闭环：

1. `DungeonScene.refreshSynergyRuntime()` 可检测激活
2. 激活时会触发 `taste heartbeat`
3. 同步发出 `synergy_activated` 事件
4. `Phase6Telemetry` 记录激活次数与首次激活楼层
5. `DomainEventEffectBinder` 会把激活写入 run log
6. save/resume 已避免恢复时重复计数

因此，“synergy 是否被触发、何时首次出现、是否在恢复后重复统计” 这类 6.0 基线问题已经收口。

### 4.2 未闭环的部分

当前缺口已经收缩到表达层：

1. `feedbackEventRouter` 还没有 `synergy activated` 的 presentation route
2. `SFXSystem` / `VFXSystem` 还没有 synergy 专属 cue
3. 玩家在战斗中仍主要通过 run log 或 run-end summary 才能知道 synergy 触发

因此 synergy 的下一步不再是“观测”，而是 **6.3 的 feedback expression**。

---

## 5. 审计结论与 6.1 交接

6.0 对 runtime effect 的价值，不是把所有系统一次做完，而是把缺口精准缩小为三件可执行的事：

1. `BuffRuntimeClosure`
   - player / monster 的 active effect state
   - apply / expire 的统一生命周期
   - buff effect 对 stats / combat 的真实消费
2. `DamageTypeResolverV1`
   - `physical / arcane` 最小分流
   - 保持 telemetry 口径不变
3. `Synergy feedback expression`
   - 保持现有 telemetry / log 基线
   - 在 6.3 增加玩家可感知的 presentation route

因此，6.0 的 runtime effect blind spot 已经关闭；  
6.1 需要做的，是把这些“已经被量化的缺口”转成真正的规则闭环。
