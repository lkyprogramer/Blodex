# Phase 6 - 6.2 战力跃迁保底、跃迁幅度与 Boss 奖励闭环

**阶段**: 6.2  
**优先级**: P0  
**对应问题**: 战力跃迁无保底 + 跃迁幅度无标准 + Boss 奖励 runtime 闭环缺失

---

## 1. 阶段目标

把“每 2 层至少 1 次跃迁”做成 runtime 合同，而不是运气现象，并确保每次跃迁的幅度达到玩家可感知阈值。

完成标准：

1. floor `1-2`、`3-4` 各至少命中 1 次 `power_spike`。
2. 所有计入合同的 spike 都达到最低幅度标准。
3. story boss 奖励链路在 runtime 中闭环。
4. spike 来源、幅度、表现与日志可追溯。

---

## 2. 当前问题

1. 现在可能出现“有 rare、无感知跃迁”。
2. 当前没有“次数合同”，更没有“幅度合同”。
3. `BossDropResult` 在 core 中存在，但实机场景未形成稳定闭环。
4. 现在很多奖励来源仍按 rarity 标签分类，而不是按“是否定义构筑”分类。

## 4. 技术方案

### 4.1 PowerSpikeBudget

1. 以 floor pair 为单位管理：
   - `1-2`
   - `3-4`
   - `5`（Boss / story reward）
2. 当前 pair 若未命中 spike，则后续通过：
   - elite reward
   - merchant
   - event reward
   - boss reward
   做补偿。
3. 保底的是“定义构筑的高价值节点”，不是固定 item id。

### 4.2 PowerSpikeAmplitudeScorer

1. 评分维度：
   - offensive delta
   - defensive delta
   - utility delta
   - TTK 变化
   - EHP / sustain 变化
2. 默认以前一刻即时快照作为基准，而不是楼层起始快照。
3. 幅度门槛：
   - offensive：`DPS >= +30%` 或 `TTK <= -25%`
   - defensive：`EHP >= +40%` 或 `sustain >= +35%`
   - 每局至少一次 `major spike >= +50%`

### 4.3 Boss Reward Runtime 闭环

推荐方案：

1. 复用现有 boss 胜利选择面板。
2. 在 `finishRun(true)` 前插入 story boss reward 结算。
3. 奖励语义与 core / simulator 对齐：
   - guaranteed rare
   - guaranteed boss-exclusive
   - 可选 bonus drop
4. abyss 分支与 boss reward 必须共享统一运行时入口，避免 story / endless 行为分裂。
5. 6.2 只完成 reward runtime 闭环与可消费事件，不在本阶段承诺独立 reward stinger；专属 cue 统一由 6.3 接手。

### 4.4 Spike Source Resolver

统一分类：

1. rare / unique 掉落
2. 技能解锁或技能关键升级
3. build threshold 达成
4. 路线 pivot
5. boss reward

只有被 `SpikeSourceResolver + AmplitudeScorer` 同时认可的事件，才计入合同 spike。

## 8. 风险与止损

1. 风险：为保底过度脚本化，损失随机性。  
   止损：保底的是节点类型，不是固定掉落。
2. 风险：幅度评分和玩家体感不一致。  
   止损：同时看数值阈值与手动录像，不只信 scorer。
3. 风险：boss reward 接入影响结算链路。  
   止损：先做 reward-close hook，再并入最终结算。
4. 风险：`DungeonScene.ts` 的历史体量债让 `check:architecture-budget` 常驻阻塞。  
   止损：本阶段追加 follow-up，将门禁收口为“目标预算 + debt ceiling 的 no-regression gate”，详见 `2026-03-06-phase6-6.2-architecture-budget-gate-followup.md`。
