# Phase 5 V2 - 5.4 架构地基收敛

**阶段**: 5.4  
**优先级**: P0  
**对应问题**: A1 + 三大超大类收敛目标

---

## 1. 阶段目标

从“RuntimeModule 假拆分”转向“显式端口 + 稳定边界”的可维护架构，避免后续再次长回 God Class。

完成标准：

1. `scenes/dungeon/*` 清零 `Record<string, any>` host 依赖。
2. `DungeonScene/MetaMenuScene/HudContainer` 达成分段压线。
3. 架构门禁可以阻止回流。

---

## 2. 当前问题

1. 多个模块通过 `this as Record<string, any>` 访问 Scene 私有状态，缺少类型保护。
2. 模块依赖图不可见，重命名/迁移风险高。
3. 三大超大类仍处于临界规模。

---

## 3. 范围与非目标

### 3.1 范围

1. `DungeonScene` 与 `scenes/dungeon/*` Host Port 改造。
2. `MetaMenuScene` 视图和动作层解耦。
3. `HudContainer` tooltip/inventory/formatter 拆分。
4. 架构检查脚本收紧。

### 3.2 非目标

1. 不新增玩法规则。
2. 不重做渲染系统。
3. 不改 content 数据定义。

---

## 4. 技术方案

### 4.1 Host Port 显式化

1. 每个 RuntimeModule 定义独立 Host 接口（最小字段集）。
2. `DungeonScene` 显式实现这些接口。
3. 模块构造函数只接收对应 Host Port，不再接收 `any`。

### 4.2 三大类瘦身策略

1. `DungeonScene`：保留装配与生命周期，业务迁移到模块/服务。
2. `MetaMenuScene`：拆 `ViewBuilder` + `ActionService`。
3. `HudContainer`：拆 `TooltipService` + `InventoryRenderService` + `Formatter`。

### 4.3 预算分段目标

1. 阶段 A：`2300 / 950 / 850`
2. 阶段 B：`1800 / 780 / 620`
3. 阶段 C（5.7 前）：`1500 / 650 / 450`
