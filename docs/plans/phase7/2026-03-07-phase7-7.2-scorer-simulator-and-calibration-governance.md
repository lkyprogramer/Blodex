# Phase 7 - 7.2 Scorer、Simulator 与 Calibration 治理

**阶段**: `7.2`  
**优先级**: `P1`  
**对应问题**: `itemTradeoff`、`PowerSpikeAmplitudeScorer`、`RealBalanceCalibration` 仍偏启发式，长期容易继续被动漂移

---

## 1. 阶段目标

把当前用于 compare、spike、evidence、balance drift 的启发式模型升级成可审计、可校准、可维护的治理体系。

完成标准：

1. `itemTradeoff` 拥有显式 calibration 资产。
2. spike amplitude scorer 拥有显式 representative sample 与阈值依据。
3. `policy threshold` 与 `calibration override` 有清晰边界。
4. heuristic / real 的差异不再主要靠人工解释。

---

## 2. 当前问题

1. `itemTradeoff` 影响 compare、merchant gating、simulator，但权重是普通常量。
2. `PowerSpikeAmplitudeScorer` 更偏 derived stats 近似，不是强战斗快照模型。
3. calibration override 已存在，但治理边界仍不够清晰。

---

## 3. 范围与非目标

### 3.1 范围

1. item tradeoff calibration
2. spike amplitude calibration
3. simulator / real diff governance
4. threshold policy split

### 3.2 非目标

1. 不在本阶段直接做完整元素体系。
2. 不在本阶段新增大型 content pool。
3. 不在本阶段新增美术/音频资源；只治理评分与校准资产。

---

## 4. 技术方案

### 4.1 `ItemTradeoffCalibrationAsset`

记录：

1. 维度权重
2. slot identity
3. sample scenarios
4. baseline artifact

### 4.2 `SpikeAmplitudeCalibrationAsset`

记录：

1. representative player snapshot
2. representative enemy snapshot
3. TTK/EHP sample
4. threshold rationale

### 4.3 `policy threshold` vs `calibration override`

规则：

1. `policy threshold` 代表长期门禁。
2. `calibration override` 代表局部样本差异吸收。
3. override 必须附 evidence artifact，不允许只留代码注释。

### 4.4 simulator / real diff 治理

1. 差异分类：
   - model incomplete
   - sampling mismatch
   - runtime bug
   - content drift
2. 只有确认不是 runtime bug，才允许加 calibration。

### 4.5 资源策略

1. 本阶段没有 gameplay 资源生产任务。
2. 如果需要 artifact，只允许新增：
   - calibration report
   - compare/spike 对照截图
   - evidence 文档引用
3. 不允许把评分治理和资源扩展混在一起做。

## 7. 出口门禁

1. compare / spike / evidence 的权重和阈值都有对应 artifact。
2. calibration 不再只是“把当前样本值编码进代码”。

---

## 8. 风险与止损

1. 风险：治理层过重。  
   止损：只覆盖直接影响 compare/spike/evidence 的模型。
2. 风险：把 runtime bug 当成 calibration 问题。  
   止损：先做 runtime bug 分类，再允许 override。
