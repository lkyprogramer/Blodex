# Phase 7 - 7.0C Evidence Registry 与 Release Closure 基建重构

**阶段**: `7.0C`  
**优先级**: `P0`  
**对应问题**: calibration / threshold / smoke / sign-off / release artifact 仍然是半手工拼装

---

## 1. 阶段目标

把 Phase 6 的 evidence / sign-off 体系从“脚本 + 文档 + 手工状态”重构成注册中心驱动的统一证据链。

完成标准：

1. calibration、threshold、smoke scenario、sign-off checklist、release artifact 都有 registry。
2. `Phase6EvidencePack` 只消费 registry 与 assessment，不再手工拼装结论。
3. smoke matrix 自动项从 assessment 派生，手动项从归档目录派生。
4. release docs 状态可被代码生成或校验。

---

## 2. 当前问题

1. calibration 已存在，但 registry 还不完整。
2. smoke matrix、sign-off checklist、release 文档之间仍可能出现口径分叉。
3. browser smoke、录像、截图、文档之间没有真正统一索引。
4. 证据链仍有“引用仓内路径但不是长期归档资产”的问题。

---

## 3. 范围与非目标

### 3.1 范围

1. `CalibrationRegistry`
2. `ThresholdRegistry`
3. `SmokeScenarioRegistry`
4. `SignoffChecklistRegistry`
5. `ReleaseArtifactIndex`
6. `Phase6EvidencePack` 改造

### 3.2 非目标

1. 不在本阶段直接调 Nightmare 数值。
2. 不在本阶段补手动录像本身；录像归档在 `7.1` 完成。
3. 不在本阶段新增 gameplay 美术/音频资源。

---

## 4. 技术方案

### 4.1 Registry 设计

必须建立五类 registry：

1. `CalibrationRegistry`
   - scenario
   - sampleSize
   - baselineCommit
   - source command
   - rationale
   - evidence artifact
2. `ThresholdRegistry`
   - policy threshold
   - scenario override
   - scope
3. `SmokeScenarioRegistry`
   - scenario id
   - automation/manual
   - source assessment / artifact
4. `SignoffChecklistRegistry`
   - checklist item
   - required evidence
   - pass/fail/pending derivation rule
5. `ReleaseArtifactIndex`
   - document path
   - screenshot / video / report path
   - owner
   - freshness / generatedAt

### 4.2 `Phase6EvidencePack` 改造

1. evidence pack 只读取 registry + assessment + artifact index。
2. 自动项状态不得硬编码。
3. baseline commit、sample size、scenario mapping 全部从 registry 来。

### 4.3 Release 文档策略

1. 文档可手写说明，但状态和证据引用必须可以校验。
2. `release-readiness`、`regression-matrix`、`taste-signoff` 要么自动生成，要么有一致性检查器。

### 4.4 Artifact 归档策略

1. browser smoke 截图、录像、report 进入统一索引。
2. 不允许只存在于某个 worktree 或临时目录。
3. release 文档必须引用归档路径，而不是源码路径替代。

### 4.5 资源策略

1. 本阶段只治理证据资产的**归档与索引**，不新增 gameplay 资源。
2. 对截图、录像、报告的约束是：
   - 路径稳定
   - 可被 release docs 引用
   - 可被 artifact index 校验

## 6. 验证方案

### 6.1 自动化

```bash
pnpm --filter @blodex/game-client test
pnpm phase6:evidence:report
pnpm ci:check
```

新增重点：

1. `evidence-registry.integrity.test`
2. `release-artifact-index.test`
3. `smoke-scenario-registry.test`
4. `signoff-checklist-registry.test`

### 6.2 手动冒烟

1. 手工检查一份 evidence pack 与对应 release 文档是否一致
2. 抽查截图、录像、报告路径是否都能打开

---

## 7. 出口门禁

1. evidence pack 不再手工拼装自动 smoke 状态。
2. release readiness / regression matrix / taste sign-off 的状态可校验。
3. artifact index 可追溯所有引用的证据资产。

---

## 8. 风险与止损

1. 风险：registry 过度设计，反而难维护。  
   止损：只注册签署真正需要的字段，不追求通用框架。
2. 风险：release 文档自动化过重。  
   止损：允许文档手写，但必须对状态与证据引用做一致性校验。

---

## 9. 与 7.1 交接

进入 `7.1` 前必须确认：

1. Nightmare 调优后的新 evidence 能被 registry 正常吸收。
2. 手动 smoke、录像和签署只需要“补证据”，不需要再改 evidence 结构。
