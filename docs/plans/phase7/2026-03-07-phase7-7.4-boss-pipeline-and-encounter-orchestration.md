# Phase 7 - 7.4 多 Boss 管线与 Encounter 编排

**阶段**: `7.4`  
**优先级**: `P2`  
**前置条件**: `7.0A ~ 7.1` 已完成并通过 `7.3` gate  
**对应问题**: 当前 Boss 管线仍天然倾向单 Boss 和单 story 流程，无法承接后续内容扩展

---

## 1. 阶段目标

把当前 Boss 运行时从“单一骨王 + 单一路径”升级为可扩展的 Boss 管线与 encounter 编排层。

完成标准：

1. Boss 选择不再绑定单一 `bone_sovereign`。
2. story boss、branch boss、challenge boss 共享统一 runtime 接口。
3. boss reward、telegraph、phase transition、drop/reward 通过统一注册表接入。

---

## 2. 当前问题

1. 当前虽然已有 `BossRuntimeModule`，但内容扩展仍然容易退回“再加一个分支”。
2. Boss encounter 与 reward / compare / telemetry 还没有完全注册化。
3. 多 Boss 一旦直接加内容，很容易把 `DungeonScene` 或 `BossRuntimeModule` 再次推胖。

---

## 3. 范围与非目标

### 3.1 范围

1. Boss registry
2. encounter dispatcher
3. boss selection policy
4. boss reward binding
5. branch / challenge / story 共用接口

### 3.2 非目标

1. 本阶段不要求一次性做完 `2~3` 个 Boss 内容。
2. 不在本阶段重做 Boss 美术。
3. 不在本阶段为单个 Boss 提前生产全量资源包；本阶段只定义管线与资源接线协议。

---

## 4. 技术方案

### 4.1 `BossRegistry`

统一注册：

1. `BossDef`
2. `encounterType`
3. `spawnPolicy`
4. `rewardPolicy`
5. `telegraphProfile`

### 4.2 `BossEncounterDispatcher`

负责：

1. 基于 floor / branch / challenge / run mode 选择 Boss
2. 注入 reward / compare / telemetry 绑定
3. 保证不同 Boss 流程复用同一套 close / reward / summary 钩子

### 4.3 reward 与 compare 绑定

1. boss reward 不再由单个 boss 模块局部拼接。
2. compare prompt / reward close / telemetry 都从统一 reward binding 走。

### 4.4 资源方案

#### 视觉方向

1. 本阶段不直接生产完整 Boss 美术，但必须定义资源接口与命名规则。
2. 所有后续 Boss 视觉资源必须遵循 [art-style-bible.md](/Users/luo/Documents/github/Blodex/docs/art-style-bible.md)：
   - `painterly-dark-fantasy-v1`
   - low-key top-left directional light
   - readable silhouette at gameplay scale
   - restrained glow / no neon
3. 必须预留以下资源位：
   - `boss_sprite`
   - `boss_portrait`
   - `telegraph_profile`
   - `boss_reward_badge`（可选）

#### 音频方向

1. 本阶段定义 Boss 音频事件键，而不是一次性生产所有音频。
2. 最小事件键集合：
   - `boss:intro`
   - `boss:phaseChange`
   - `boss:telegraph`
   - `boss:defeat`
   - `boss:reward`
3. 音频风格延续现有 `grim-fantasy-audio-v1`，要求：
   - 低频压迫
   - 金属/骨骼/石质打击感
   - 避免合成器感和高亮电子音

#### 资产治理

1. 视觉资源先写入 `assets/source-prompts/asset-plan.yaml`，通过 `assets:validate`。
2. 音频资源先写入 `assets/source-prompts/audio-plan.yaml`，通过 `assets:audio:validate`。
3. 资源命名要预留多 Boss 批次扩展，不允许先写死 `bone_sovereign` 风格。
4. 本阶段若需要实际生成 Boss 视觉资源，必须先由用户提供 Gemini Key；未提供前只允许冻结 prompt、manifest 入口与占位资源。
5. 在 Gemini Key 未提供前，不允许直接运行 Boss sprite / portrait / telegraph 的生成步骤。
