# Phase 7 - 7.0A Scene Shell、UI Shell 与 Meta Shell 重构

**阶段**: `7.0A`  
**优先级**: `P0`  
**对应问题**: `DungeonScene / HudContainer / MetaMenuScene / MetaMenuPanel` God Object 债、Host Port 膨胀、UI 业务语义回流

---

## 1. 阶段目标

把当前客户端最重的三个聚合点真正拆成 shell + facade + controller 结构，而不是继续用 debt ceiling 冻结历史体量。

完成标准：

1. `DungeonScene` 降为真正的 `Scene Shell`，只保留生命周期、顶层装配、帧入口与 fatal boundary。
2. `HudContainer` 只保留渲染与容器职责，不再承载 compare / overlay / summary / toast 的业务语义。
3. `MetaMenuScene` 降为导航与装配层；`MetaMenuPanel` 降为纯 view 组合层。
4. `check:architecture-budget` 对上述文件不再依赖 debt ceiling。
5. 新边界通过 typed facade / typed port 显式表达，禁止继续使用“场景大对象直通访问”。

---

## 2. 当前问题

1. `DungeonScene` 仍承担 run、reward、feedback、save、debug、UI bridge 等多重职责。
2. `HudContainer` 仍吞掉 compare prompt 的动作语义、overlay blocking 语义和部分 summary 行为。
3. `MetaMenuScene` 与 `MetaMenuPanel` 之间没有清晰的业务 / 视图边界。
4. 当前的“拆分”很多是函数迁出，不是状态边界与协调边界迁出。
5. 主干虽然能过 architecture budget，但依赖的是 no-regression debt ceiling，不是目标结构达标。

---

## 3. 范围与非目标

### 3.1 范围

1. `DungeonScene` 壳层化。
2. `HudContainer` 容器化。
3. `MetaMenuScene / MetaMenuPanel` 分层。
4. 新 facade / coordinator / controller 边界与装配方式。
5. architecture budget 硬门禁恢复。

### 3.2 非目标

1. 不在本阶段新增新 Boss、新 affix、新元素系统。
2. 不在本阶段重新设计玩法规则。
3. 不在本阶段重做 HUD 视觉风格。
4. 不在本阶段改动 save schema 本身；save 重构属于 `7.0B`。
5. 不在本阶段新增美术/音频资源；UI shell 拆分必须复用现有视觉语言。

---

## 4. 技术方案

### 4.1 `DungeonScene` 重构目标结构

引入以下层次：

1. `DungeonScene`
   - Phaser scene shell
   - lifecycle
   - dependency wiring
   - frame entry
2. `DungeonSessionFacade`
   - 聚合 runtime state surfaces
   - 对下游暴露精简 typed host
3. `RunFlowCoordinator`
   - 统一 active frame / event panel / boss victory / post-run update 分支
4. `RewardAndCompareCoordinator`
   - loot / merchant / boss reward / compare queue / reward-close hook
5. `HeartbeatFeedbackCoordinator`
   - heartbeat queue、toast budget、feedback priority、overlay-safe dispatch
6. `DungeonDiagnosticsFacade`
   - diagnostics snapshot、debug export、evidence debug view

要求：

1. `DungeonScene` 不得直接保存 compare queue、reward queue、overlay semantics。
2. runtime module 不再直接依赖 `DungeonScene` 整体，而依赖 facade 暴露的最小 host。
3. Scene 里只允许“装配”和“调用 facade”，不允许继续长出业务分支。

### 4.2 `HudContainer` 重构目标结构

拆成如下 controller / presenter：

1. `OverlayCoordinator`
   - 统一 event panel、compare prompt、summary、boss reward overlay 的 blocking contract
2. `EquipmentCompareController`
   - prompt state、button action、defer/ignore/equip typed action
3. `HeartbeatToastController`
   - toast queue、priority、replace / coalesce 策略
4. `RunSummaryController`
   - end-run summary、next-run suggestions、report binding
5. `CombatHudPanelController`
   - 战斗 HUD 面板组合与数据映射

要求：

1. `HudContainer` 只渲染视图组件，不定义 `equip/later/ignore` 的业务后果。
2. 所有 action 必须通过 typed callback 回到 controller / runtime。
3. compare / toast / summary 的状态机不得再分散在多个 UI 组件内部。

### 4.3 `MetaMenuScene` / `MetaMenuPanel` 重构目标结构

拆成：

1. `MetaMenuScene`
   - route / navigation
   - scene lifecycle
   - dependency wiring
2. `MetaFlowController`
   - continue run
   - blueprint / unlock flow
   - tab switching policy
   - meta action dispatch
3. `MetaMenuPanel`
   - 纯视图容器
4. 子 view
   - `MetaBlueprintView`
   - `MetaProgressView`
   - `MetaContinueRunView`
   - `MetaTabsView`

要求：

1. Scene 不再直接拼装全部面板状态。
2. Panel 不再直接驱动业务副作用。

### 4.4 Host Port 收缩策略

1. 所有新 facade 暴露 typed host，不允许 `Record<string, unknown>` 或裸场景 cast。
2. 每个 coordinator/controller 的 host 只覆盖本职责最小字段。
3. 先建 facade，再把旧模块逐步改成依赖 facade host。

### 4.5 预算门禁恢复策略

1. 删除 `DungeonScene / HudContainer / MetaMenuScene / MetaMenuPanel` 的 debt ceiling。
2. 使用三段式硬检查点：
   - `PR-7.0A-01`: `DungeonScene <= 2800`，`HudContainer` no-growth
   - `PR-7.0A-02`: `DungeonScene <= 1500`，`HudContainer <= 650`
   - `PR-7.0A-03`: `MetaMenuScene <= 650`，`MetaMenuPanel <= 450`，`HudContainer <= 450`
3. 任一检查点未达成，不得进入下一 PR。

### 4.6 资源策略

1. 本阶段默认 **不新增** 美术或音频资源。
2. 如拆分 UI 后发现必须补最小占位资源，只允许：
   - 复用现有 `tokens.css`、现有 prompt frame、现有 VFX/SFX key
   - 或使用纯布局/颜色/排版调整解决
3. 禁止把“UI shell 拆分”与“视觉换皮”绑定。
4. 如确有不可避免的 UI 资产缺口，必须遵循 [art-style-bible.md](/Users/luo/Documents/github/Blodex/docs/art-style-bible.md)，但应延后到独立 follow-up，而不是混入本阶段主 PR。
5. 即使识别出未来需要补 UI 美术，也只能先冻结 prompt / 资源位；在用户提供 Gemini Key 前，不允许直接运行任何图片生成步骤。

---

## 8. 风险与止损

1. 风险：只拆文件，不拆状态与协调边界。  
   止损：中间检查点同时要求职责迁移，不只看行数。
2. 风险：UI/controller 反而引入更多转发层。  
   止损：只保留有独立状态机价值的 controller，不为了“层次整齐”盲拆。
3. 风险：Scene 拆分过快导致运行链回归。  
   止损：按 active frame、reward path、overlay path 三条主链逐步迁移。

---

## 9. 与 7.0B 交接

进入 `7.0B` 前必须确认：

1. save/restore 不再依赖 `DungeonScene` 作为大状态容器。
2. compare / reward / overlay 语义已经迁出 UI 容器层。
3. 新 facade 已能为 save pipeline 提供稳定的 typed surfaces。
