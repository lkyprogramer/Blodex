# Phase 7 - 7.0B RunSaveV3 与 Restore Pipeline 重建

**阶段**: `7.0B`  
**优先级**: `P0`  
**对应问题**: save/resume 状态层次混乱、timeline rebasing 脆弱、旧兼容路径与 fallback 逻辑污染结构

---

## 1. 阶段目标

用全新的严格 schema 和恢复流水线替换现有 save/resume 拼接式实现，彻底消化 Phase 6 期间暴露出来的恢复脆弱性。

完成标准：

1. 引入全新 `RunSaveV3` 严格 schema。
2. restore 只允许经过 `strict deserialize -> load -> derived rebuild -> ephemeral bootstrap`。
3. 删除旧 save key、旧 normalize helper、旧 fallback deserialize。
4. compare queue、prompt state、buff timeline、reward transient state 全部显式归类。
5. 对旧 save key 不做 silent wipe，而是给出一次性提示后清理。

---

## 2. 当前问题

1. save/resume 现在混杂了 persistent state、derived state、session-local state、UI transient state。
2. 过去多个阶段都在持续修 restore 问题，说明模型边界不稳定。
3. timeline 字段（`nowMs / expiresAtMs / appliedAtMs / queue pointer`）没有统一分类。
4. 旧兼容路径会让重构后的结构继续背负历史条件分支。

---

## 3. 范围与非目标

### 3.1 范围

1. `RunSaveV3`
2. serializer / deserializer
3. restore pipeline
4. derived rebuild / ephemeral bootstrap
5. save key 策略与用户提示

### 3.2 非目标

1. 不保留旧 save 兼容。
2. 不做旧数据迁移。
3. 不在本阶段扩玩法系统。
4. 不新增美术/音频资源；旧存档重置提示必须复用现有 UI/toast 能力。

---

## 4. 技术方案

### 4.1 新状态分层

强制拆成四层：

1. `Persistent Domain State`
   - run/player/equipment/inventory/currency/route/mutation
2. `Persistent Runtime Snapshot`
   - floor progression、boss runtime core state、event node core state、spike budget、telemetry aggregates
3. `Restore-Derived State`
   - buff-derived player stats
   - monster derived moveSpeed / damage profile
   - compare candidate baselines
4. `Ephemeral Session State`
   - overlay lock
   - compare queue pointer
   - active prompt context
   - session-local timers

### 4.2 `RunSaveV3` 规则

1. 新 schema 严格定义每个字段归属。
2. 不保留 `V1/V2` 类型与 helper。
3. 所有旧 key 视为过期格式。

### 4.3 Restore Pipeline

固定顺序：

1. strict deserialize
2. load persistent data into state containers
3. run derived rebuilders
4. bootstrap ephemeral runtime
5. hand control back to `DungeonSessionFacade`

禁止：

1. 在 deserialize 阶段直接 mutate Scene
2. 在 derived rebuild 中读取 UI transient state
3. 用 fallback 魔法值重建运行时

### 4.4 Timeline / Buff / Queue 规则

1. runtime local 时间字段只允许保存在会话态或以剩余时长形式保存。
2. buff 只保存剩余时长或稳定定义，不保存跨 session 的绝对 runtime 时间。
3. compare / reward / prompt queue 必须有显式 snapshot 与 restore 语义。

### 4.5 旧 key 清理策略

1. 若检测到旧 save key 存在且 `RunSaveV3` 不存在：
   - 显示一次性提示：`存档格式已重置，旧进度不可继续`
   - 记录一次 release-note style log
   - 清理旧 key
2. 不尝试迁移旧数据。

### 4.6 资源策略

1. 本阶段不新增 UI 贴图、图标、音频 cue。
2. “存档格式已重置，旧进度不可继续” 必须复用现有 toast / modal 样式与现有音频体系。
3. 不允许为了 save 重构单独引入新的提示资产。

---

## 8. 风险与止损

1. 风险：重建 schema 时遗漏 transient 状态。  
   止损：先列完整状态清单，再实现 serializer。
2. 风险：derived rebuild 顺序错误导致恢复后属性异常。  
   止损：所有 rebuild stage 独立测试，不允许隐式顺序。
3. 风险：完全破坏式修改影响玩测节奏。  
   止损：用一次性提示 + release note 减少困惑，但不做旧数据迁移。
